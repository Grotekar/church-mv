# AGENTS.md

Статус: Approved

## Назначение

Этот файл определяет правила работы ИИ-ассистентов с данным репозиторием.

Он не заменяет проектную документацию и не является самостоятельным источником продуктовых, дизайнерских, контентных или архитектурных решений.

Документация проекта имеет приоритет.

---

## Область действия

AGENTS.md распространяется на:

- Codex;
- ChatGPT;
- Claude;
- другие ИИ-ассистенты, работающие с репозиторием.

Документ не распространяется на людей-разработчиков.

Он определяет только рекомендуемое поведение ИИ при работе с проектом.

---

## Порядок изучения документации

Перед выполнением любой задачи ИИ должен ознакомиться с документами в следующем порядке:

1. `01-Видение.md`
2. `02-Принципы-проекта.md`
3. `03-Дизайн-контракт.md`
4. `04-Контент-контракт.md`
5. `05-Техническая-архитектура.md`
6. `06-Архитектура-контента.md`

Если задача относится только к одной области, допускается изучить соответствующий документ повторно.

---

## Источник истины

Документация проекта является единственным источником истины.

Если реализация противоречит утверждённой документации, необходимо считать, что реализация требует исправления.

Не изменять документацию только потому, что код написан иначе.

---

## Приоритет документов

При конфликте документов используется следующий приоритет:

1. Видение.
2. Принципы проекта.
3. Контракты.
4. AGENTS.md.

AGENTS.md никогда не должен переопределять утверждённые проектные документы.

---

## Перед началом работы

Перед выполнением любой задачи ИИ обязан определить:

- к какой области относится изменение;
- какой контракт регулирует данную область;
- не противоречит ли предполагаемое решение существующим документам.
- требуется ли изменение существующей документации.

Если существующие документы полностью покрывают задачу, сначала следует выполнить реализацию.

Если задача выходит за рамки утверждённой документации, сначала необходимо предложить изменение документации.

Если подходящий документ отсутствует или возникает противоречие между документами, ИИ должен остановиться и запросить уточнение.

---

## Во время работы

ИИ обязан:

- соблюдать утверждённую архитектуру;
- соблюдать утверждённый дизайн;
- соблюдать правила контента;
- соблюдать принципы проекта;
- использовать существующие архитектурные решения.

Не следует предлагать изменения философии проекта без отдельного запроса.

---

## Запрещается

ИИ не должен:

- придумывать факты;
- добавлять новые функции без запроса;
- менять структуру страниц без обсуждения;
- самостоятельно изменять архитектуру;
- использовать placeholder как реальные данные;
- добавлять зависимости без необходимости;
- использовать шаблонные решения только потому, что они популярны.

---

## Принцип минимального вмешательства

При выполнении задачи ИИ должен изменять только те файлы, которые действительно относятся к поставленной задаче.

Не следует:

- выполнять попутный рефакторинг;
- исправлять стиль несвязанных файлов;
- менять структуру проекта без необходимости.

---

## При внесении изменений

Предпочитать минимально необходимые изменения.

Не выполнять крупный рефакторинг, если задача этого не требует.

Не изменять несвязанный код.

Не менять форматирование файлов без необходимости.

Не переименовывать компоненты только ради единообразия.

---

## После завершения задачи

Перед завершением работы ИИ должен проверить:

- соответствует ли изменение всем утверждённым контрактам;
- не появились ли placeholder;
- не нарушена ли архитектура;
- не ухудшилась ли доступность;
- не ухудшилась ли производительность;
- не появились ли лишние зависимости.

Перед завершением работы ИИ должен определить, требует ли выполненное изменение обновления проектной документации.

Если документация устарела вследствие осознанного изменения проекта, ИИ должен сообщить об этом.

Если документация остаётся актуальной, изменять её не следует.

Если возможно:

- выполнить lint;
- выполнить build.

---

## Неуверенность

Если ИИ не уверен:

- в фактах;
- в архитектурном решении;
- в требованиях;
- в богословской формулировке;
- в интерпретации документации,

необходимо остановиться и запросить уточнение.

Недостаток информации никогда не должен компенсироваться предположениями.

---

## Формат ответа

После завершения задачи ИИ должен предоставить краткий отчёт.

Структура отчёта:

Изменено:

- ...

Создано:

- ...

Удалено:

- ...

Проверки:

- lint
- build

Замечания:

- ...

Отчёт должен быть кратким и содержать только фактическую информацию.

---

## Если документации недостаточно

Если задача требует изменения, которое не описано существующими контрактами, ИИ не должен самостоятельно принимать архитектурное, дизайнерское или продуктовое решение.

Вместо этого необходимо предложить создать или обновить соответствующий документ, а затем выполнять реализацию только после его утверждения.
