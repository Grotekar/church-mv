# Документация проекта

Каталог `docs` содержит проектную документацию сайта Церкви Божией города Минеральные Воды.

Документация проекта определяет не только реализацию сайта, но и принципы принятия решений при его дальнейшем развитии. Перед внесением существенных изменений рекомендуется ознакомиться с соответствующими документами.

---

# Как читать документацию

Документацию следует читать от общего к частному: сначала понять назначение сайта и принципы проекта, затем перейти к конкретным контрактам.

## Уровень 1. Видение проекта

`01-Видение.md`

Отвечает на вопрос:

«Зачем существует сайт?»

Этот документ задаёт смысл проекта и определяет, каким сайт должен быть на уровне назначения.

---

## Уровень 2. Принципы проекта

`02-Принципы-проекта.md`

Определяет правила принятия решений.

Этот документ помогает выбирать между разными вариантами развития проекта, когда возможны несколько подходов.

---

## Уровень 3. Контракты

`03-Дизайн-контракт.md`

Определяет визуальную философию проекта.

`04-Контент-контракт.md`

Определяет требования к текстам.

`05-Техническая-архитектура.md`

Определяет технические решения.

`06-Архитектура-контента.md`

Определяет организацию хранения контента.

Контракты фиксируют утверждённые решения в конкретных областях: дизайн, тексты, техническая архитектура и структура контента.

---

## Уровень 4. Работа ИИ

`AGENTS.md`

Определяет правила работы ИИ-ассистентов с проектом.

Этот файл находится в корне репозитория и объясняет, как ИИ должен использовать проектную документацию при выполнении задач.

---

## Уровень 5. Управление проектом

Планируются документы:

`08-План-развития.md`

Будет описывать очередность развития сайта и приоритеты будущих изменений.

`09-ADR.md`

Будет фиксировать важные архитектурные решения, причины их принятия и последствия для проекта.

`00-Глоссарий.md`

Будет определять официальную терминологию проекта и обеспечивать единообразное использование терминов во всей документации и интерфейсе.

Каждый следующий уровень опирается на предыдущий. Чем выше документ в схеме, тем более общий и приоритетный смысл он задаёт.

---

# Принцип работы

Изменения проекта выполняются по следующему жизненному циклу:

Возникла идея

↓

Проверяется существующая документация

↓

Если документации достаточно — реализуется изменение

↓

Если документации недостаточно — сначала обновляется документация

↓

После утверждения документации выполняется реализация

↓

Проводится проверка

Такой порядок помогает не превращать проект в набор случайных решений. Сначала фиксируется основание для изменения, затем меняется реализация.

---

# Когда обновляется документация

Документация обновляется, если:

- меняется философия проекта;
- принимается новое архитектурное решение;
- появляются новые постоянные правила;
- существующий документ перестал отражать утверждённое состояние проекта.

Не следует изменять документацию:

- ради временных экспериментов;
- потому что реализация ей противоречит;
- ради косметических правок без изменения смысла.

---

# Статусы документов

В проекте используются следующие статусы:

- Draft — документ обсуждается и ещё не является утверждённым источником истины.
- Approved — документ утверждён и обязателен для дальнейшей работы.
- Deprecated — документ больше не используется как актуальное правило.
- Archived — документ сохранён только для истории.

Статус помогает понять, можно ли опираться на документ при принятии решений.

---

# Иерархия документов

При конфликте документов используется следующий порядок приоритета:

1. Видение.

2. Принципы проекта.

3. Контракты.

4. AGENTS.md.

5. Реализация.

Если возникает конфликт, приоритет имеет документ более высокого уровня.

Реализация не является источником истины, если она противоречит утверждённой документации.

---

# Для кого предназначена документация

Документация предназначена для:

- разработчиков;
- ИИ-ассистентов;
- будущих сопровождающих проекта;
- участников церкви, принимающих архитектурные решения.

Она помогает сохранять последовательность проекта даже при смене исполнителей или длительных паузах в разработке.

---

# Принципы сопровождения

Основные принципы сопровождения:

- документация важнее реализации;
- решения принимаются осознанно;
- архитектура развивается постепенно;
- предпочтение отдаётся простым решениям;
- любые изменения должны иметь понятную причину.

Если изменение не имеет ясного основания в документации, сначала необходимо обсудить и зафиксировать это основание.

---

# Создание новых документов

Новый документ создаётся только тогда, когда существующие документы уже не позволяют ясно описать новую область проекта.

Предпочтительно расширять существующие документы, а не увеличивать количество файлов.

Новый документ должен иметь понятную область действия и место в общей системе документации.
