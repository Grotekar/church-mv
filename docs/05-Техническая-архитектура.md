## Назначение

Документ определяет технические принципы реализации сайта Церкви Божией города Минеральные Воды.

Он описывает архитектурные решения проекта, причины их выбора и границы допустимых изменений. Его задача — не усложнить разработку, а зафиксировать простой и устойчивый технический подход, который можно поддерживать одним разработчиком.

После утверждения этого документа любые изменения архитектуры должны быть осознанным решением, а не случайным результатом очередной задачи.

---

## Область действия

Документ распространяется на:

- структуру проекта;
- технологии;
- сборку;
- размещение контента;
- организацию компонентов;
- организацию страниц.

Документ не регулирует:

- внешний вид;
- тексты;
- вероучение.

Для этих областей существуют отдельные утверждённые документы.

# 1. Архитектурные цели

Архитектура проекта должна поддерживать следующие цели:

- простота;
- долговечность;
- минимальная зависимость от сторонних библиотек;
- высокая производительность;
- лёгкость сопровождения одним разработчиком;
- возможность статического размещения;
- понятное добавление новых страниц и материалов.

Главная техническая задача — сделать сайт достаточно простым, чтобы его можно было понимать, изменять и публиковать без сложной инфраструктуры.

Архитектура не должна создавать ощущение большого продукта там, где нужен небольшой, аккуратный и надёжный сайт поместной церкви.

---

# 1.1. Архитектурные границы

В рамках версии 1.x архитектура сознательно не включает:

- CMS;
- серверную часть;
- базу данных;
- авторизацию;
- административную панель;
- формы с серверной обработкой;
- поиск;
- локализацию;
- личные кабинеты;
- интеграции со сторонними сервисами, кроме обычных внешних ссылок.

Отсутствие этих возможностей является осознанным архитектурным решением, а не ограничением проекта.

Такие возможности могут рассматриваться в будущем, но их появление требует отдельного архитектурного решения и обновления документации.

---

# 2. Основной стек

Проект использует следующий стек:

- Next.js App Router;
- React;
- TypeScript;
- Tailwind CSS;
- Static Export.

Next.js App Router выбран как современная и распространённая основа для страниц, маршрутов, metadata и статической генерации.

React используется как базовая модель компонентов. Он позволяет собирать страницы из понятных частей без введения дополнительного UI-фреймворка.

TypeScript выбран для явного описания структуры контента, props компонентов и снижения риска случайных ошибок при изменении сайта.

Tailwind CSS используется как простой способ оформлять интерфейс без отдельной сложной дизайн-системы и без подключения UI-библиотек.

Static Export выбран потому, что сайт должен работать без серверного кода и размещаться на обычном shared hosting.

# 3. Принципы архитектуры

Основные принципы архитектуры:

- минимальное количество зависимостей;
- предпочтение встроенных возможностей платформы;
- простая структура проекта;
- отсутствие преждевременных абстракций;
- отсутствие сложных паттернов без необходимости;
- разделение контента и представления;
- предсказуемость сборки и публикации.

Если задачу можно решить средствами Next.js, React, TypeScript, CSS и HTML, не следует добавлять новую библиотеку.

Архитектура должна оставаться понятной при чтении файлов проекта, без необходимости изучать внутренние соглашения, которые не дают реальной пользы.

---

# 4. Организация проекта

Основные каталоги проекта:

`app/` — маршруты, страницы, layout, metadata и другие файлы App Router.

`components/` — переиспользуемые компоненты интерфейса, которые действительно упрощают чтение страниц.

`content/` — структурированные данные и тексты сайта. Этот каталог является основным местом редактирования церковного и событийного контента до появления CMS.

`public/` — статические файлы, доступные напрямую: изображения, логотип, QR-код, favicon и другие публичные ассеты.

`docs/` — утверждённая проектная документация и документы в статусе Draft.

`lib/` — небольшие вспомогательные функции, если они нужны нескольким частям проекта.

Новые каталоги добавляются только тогда, когда существующая структура перестаёт ясно описывать назначение файлов.

---

# 4.1. Конфигурация проекта

Конфигурация проекта должна быть централизованной.

Не следует дублировать одни и те же значения в нескольких местах.

Публичные настройки сайта должны храниться отдельно от реализации компонентов.

Настройки сборки должны быть минимальными и понятными.

Если настройка влияет на поведение или публикацию сайта, её назначение должно быть очевидно разработчику, который будет сопровождать проект позже.

---

# 5. Организация компонентов

Компоненты должны быть:

- небольшими;
- переиспользуемыми;
- понятными.

Компонент создаётся, если он:

- упрощает чтение страницы;
- повторяется в нескольких местах;
- отделяет самостоятельную часть интерфейса;
- снижает риск ошибок при изменении похожих блоков.

Не создавать компонент только ради абстракции.

Если компонент используется один раз и не упрощает код, допускается оставить разметку внутри страницы.

Компоненты не должны становиться местом хранения церковного контента. Тексты, ссылки, адреса, расписание и другие данные должны приходить из `content/`.

---

# 5.1. Маршрутизация

Структура URL должна быть простой и предсказуемой.

Маршруты должны отражать структуру сайта, а не внутреннюю реализацию компонентов или временные технические решения.

URL не должны зависеть от того, как именно организован код внутри страницы.

Изменение URL считается архитектурным изменением, потому что оно влияет на навигацию, SEO, внешние ссылки и долгосрочную доступность страниц.

---

# 6. Работа с контентом

До появления CMS весь контент хранится в проекте.

Контент отделён от компонентов.

Компоненты не содержат текстов.

Контент не содержит JSX.

Контент должен храниться в простой и редактируемой форме: TypeScript-объекты, массивы и строки.

Структура контента должна быть достаточно явной, чтобы волонтёр-разработчик мог быстро найти и изменить нужное значение.

Если данные неизвестны, используется placeholder с явным TODO-комментарием, либо информация не публикуется.

Контент не должен зависеть от структуры компонентов.

Изменение компонентов не должно требовать изменения структуры контента без объективной необходимости.

Это помогает сохранять долгосрочную совместимость между реализацией и данными.

---

# 7. Работа со статикой

Статические файлы размещаются в `public/`.

К статике относятся:

- изображения;
- логотип;
- QR-код;
- фотографии;
- favicon;
- файлы, которые должны быть доступны напрямую после сборки.

Изображения должны использоваться только тогда, когда они поддерживают утверждённый контент и не создают ложного впечатления о церкви.

Логотип должен быть необязательным: интерфейс обязан корректно работать как с логотипом, так и без него.

QR-код пожертвования публикуется только при наличии реального подтверждённого файла.

Фотографии должны быть реальными материалами церкви или мероприятий. Стоковые изображения не используются.

---

# 8. Зависимости

Перед добавлением новой библиотеки необходимо ответить:

- какую проблему она решает;
- можно ли решить задачу средствами платформы;
- кто будет сопровождать эту зависимость через несколько лет.

Новая зависимость допустима только тогда, когда она заметно снижает сложность, решает реальную задачу и не противоречит цели долгосрочной поддержки.

Не добавлять зависимости ради удобства одной небольшой задачи, визуального эффекта или привычного паттерна.

---

# 9. Производительность

Проект должен предпочитать:

- статическую генерацию;
- оптимизированные изображения;
- минимальный JavaScript;
- быструю загрузку;
- простую HTML-структуру;
- ограниченное количество интерактивности.

Не добавлять тяжёлые библиотеки без необходимости.

Интерактивность должна использоваться только там, где она помогает пользователю получить информацию или выполнить понятное действие.

Производительность важна не как отдельная метрика, а как часть уважительного отношения к посетителям сайта, включая людей с медленным интернетом или недорогими устройствами.

---

# 9.1. Обработка ошибок

Если данные отсутствуют, интерфейс должен деградировать корректно.

Отсутствие второстепенной информации не должно ломать страницу.

Placeholder не должен попадать в опубликованную версию сайта.

Если без данных невозможно корректно показать раздел, предпочтительнее не выводить этот раздел, чем публиковать недостоверную или вводящую в заблуждение информацию.

---

# 10. Доступность

Техническая реализация должна поддерживать доступность интерфейса.

Необходимо использовать:

- семантическую разметку;
- корректные `aria`-атрибуты только там, где они действительно нужны;
- keyboard navigation для интерактивных элементов;
- понятные focus-состояния;
- корректное управление фокусом там, где интерфейс меняет состояние.

Если элемент выглядит как действие, он должен быть реализован как доступное действие.

Если элемент является ссылкой, следует использовать `<a>`. Если элемент выполняет действие на странице, следует использовать `<button>`.

---

# 11. Статический хостинг

Проект должен корректно работать на обычном shared hosting.

Архитектура не должна зависеть от серверного кода.

Основной сценарий публикации — загрузка содержимого папки `out/` на хостинг.

Недопустимо добавлять обязательные runtime-зависимости от Node.js-сервера, базы данных, serverless-функций или API, если такое решение не было отдельно утверждено.

Сайт должен оставаться работоспособным как набор статических файлов.

---

# 11.1. Совместимость

Архитектура должна сохранять совместимость:

- с современными браузерами;
- с мобильными устройствами;
- со статическим экспортом;
- с обычным shared hosting.

Не использовать решения, ухудшающие эту совместимость без серьёзной причины.

Если новое техническое решение требует особых условий размещения или ограничивает доступность сайта на обычных устройствах, оно требует отдельного архитектурного рассмотрения.

---

# 12. Будущее развитие

Допустимые изменения без пересмотра архитектуры:

- новые страницы;
- новые мероприятия;
- новые разделы;
- новые фотографии;
- обновление текстов;
- обновление контактных данных;
- добавление новых записей в существующие структуры контента.

Изменения, требующие пересмотра архитектуры:

- CMS;
- backend;
- база данных;
- авторизация;
- поиск;
- API;
- личные кабинеты;
- формы с серверной обработкой;
- сложная система ролей или администрирования.

Такие изменения могут быть полезны в будущем, но они должны рассматриваться как отдельные архитектурные решения, а не появляться постепенно и незаметно.

Новые архитектурные возможности должны вводиться постепенно.

Предпочтительно развивать существующую архитектуру, а не заменять её полностью.

Крупные изменения, например переход на CMS или серверную архитектуру, должны оформляться отдельным архитектурным решением и сопровождаться обновлением документации.

---

# 13. Архитектурный чек-лист

Перед завершением любой технической задачи рекомендуется проверить:

- не нарушена ли структура каталогов;
- не появились ли лишние зависимости;
- не смешался ли контент с компонентами;
- не ухудшилась ли поддерживаемость;
- не нарушены ли предыдущие архитектурные решения.

Это не обязательная процедура публикации, а инструмент самопроверки разработчика.

---

# Критерий успешности архитектуры

Архитектура считается успешной, если проект остаётся понятным одному разработчику спустя несколько лет и новые страницы можно добавлять без усложнения существующего кода.

Хорошая архитектура в этом проекте не стремится быть универсальной. Она должна быть достаточно простой, чтобы поддерживать реальные задачи сайта и не создавать лишнюю нагрузку на церковь.

Документ не должен изменяться только потому, что текущая реализация ему не соответствует.
