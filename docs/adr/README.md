# Architecture Decision Records

ADR — это запись архитектурного решения.

Такие документы используются для фиксации важных решений, которые влияют на устройство проекта, его сопровождение и дальнейшее развитие.

ADR не заменяют основные проектные контракты. Они объясняют, почему было принято конкретное архитектурное решение и какие последствия оно имеет.

---

## Связь с проектной документацией

ADR не заменяют контракты проекта.

Контракты определяют утверждённые правила и границы проекта. ADR фиксируют причины конкретных архитектурных решений внутри этих границ.

При конфликте между ADR и утверждёнными контрактами приоритет имеют утверждённые документы.

Новый ADR не должен противоречить существующим контрактам без их предварительного изменения.

---

## Зачем используются ADR

ADR помогают:

- сохранять историю принятых решений;
- понимать причины архитектурных ограничений;
- не возвращаться к одним и тем же спорам без новых оснований;
- отделять осознанные решения от случайных изменений реализации.

---

## Когда создавать новый ADR

Новый ADR создаётся, если принимается решение, которое влияет на:

- архитектуру проекта;
- способ хранения контента;
- сборку и публикацию;
- зависимости;
- инфраструктуру;
- долгосрочную поддержку сайта.

ADR не нужен для мелких правок, исправления опечаток, обновления текстов или локальных изменений, которые не меняют архитектурный подход.

---

## Нумерация

ADR нумеруются последовательно:

- `ADR-0001`;
- `ADR-0002`;
- `ADR-0003`;
- и далее.

Номер ADR не меняется после создания документа.

---

## Изменение решений

Уже существующие ADR не переписываются задним числом.

Если решение устарело, отменяется или изменяется, создаётся новый ADR, который явно указывает, какое предыдущее решение он заменяет или уточняет.

Это позволяет сохранить историю проекта и понять, почему архитектура развивалась именно так.
